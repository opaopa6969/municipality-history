# 設計判断: reason 列の自由文解析方針

## 背景

`MunicipalityChange.reason` 列は e-Stat の「改正事由」フィールドをそのまま収録した自由文です。
この列には合併に関わった旧自治体名と旧コードが記載されており、機械的に解析できれば
「このコードから何に合併されたか」「この合併に参加した自治体はどれか」を
プログラムで取得できます。

---

## reason 列のフォーマット

実際のデータを観察すると、以下のパターンが存在します。

### パターン A: 「が〜に市制施行」

```
石狩町(01303)が石狩市(01215)に市制施行
```

- 旧自治体名（コード）が 1 件
- 新自治体名（コード）が 1 件

### パターン B: 「が合併し〜を新設」

```
城辺町(47371)、下地町(47372)、上野村(47373)、伊良部町(47374)、平良市(47206)が合併し、宮古島市(47214)を新設
```

- 旧自治体名（コード）が複数件（読点区切り）
- 新自治体名（コード）が 1 件

### パターン C: 「が〜に編入合併」

```
厚田村(01304)、浜益村(01305)が石狩市(01215)に編入合併
```

- 旧自治体名（コード）が複数件
- 吸収先自治体名（コード）が 1 件

### パターン D: コードなし・略称

稀に自治体コードが省略されているケースや、「等」でまとめられているケースがあります。

---

## 現バージョンの方針: 解析なし（文字列そのまま）

現バージョン (1.0.0) では `reason` 列を **解析せず文字列として保持** します。

理由:

1. **誤解析リスク** — 自由文のフォーマットは完全には統一されていません。正規表現で処理すると一部ケースで誤った結果を返す可能性があります。
2. **API 安定性** — 解析結果を返す API を公開すると、後から修正したときに breaking change になります。
3. **十分な利用価値** — `reason` をそのまま表示・ログ出力するだけで実用になるケースが大半です。

---

## `findByName` への影響

`reason` 列を検索対象に含めている `findByName` は、この非構造化テキストを部分一致検索します。
例: `findByName("石狩市")` は石狩市のレコードだけでなく、
`reason` 中に「石狩市(01215)に編入合併」と書かれた他自治体のレコードも返します。

これは **設計上の意図した動作** です（関連する変遷をまとめて取得できる）が、
名称のみで絞り込みたい場合は呼び出し側でフィルタしてください。

```java
// reason にヒットしたレコードを除外して名称のみで絞り込む
history.findByName("石狩市").stream()
    .filter(c -> c.fullName().contains("石狩市"))
    .toList();
```

---

## 将来の拡張候補

バックログに記録されている以下の機能は、構造化パースを前提とします。

- **旧コード → 新コード変換テーブル** — パターン A/B/C から `(旧コード, 新コード)` ペアを生成
- **合併ネットワーク** — 複数の廃置分合を有向グラフで表現し、「どこに吸収されたか」を辿れる API

実装する場合は、まずパターン A〜C のカバレッジを測定し、
カバーできないケースの件数・割合を確認してから API 設計に進むことを推奨します。
